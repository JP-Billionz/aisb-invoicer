# Deploy Checklist — AISB Invoicer Phase 1A → Render (free) + Neon (free)

Estimated time: **10–15 minutes** (Neon DB + Blueprint apply + first build).
No card on file required for staging.

## What's already in the repo

- `prisma/migrations/20260517000000_init/migration.sql` — initial schema migration
- `render.yaml` — Blueprint that defines the **Web Service only**
  (free plan, Ohio). Postgres is **external** because Render no longer
  offers a free Postgres tier.
- Build command: `npm install && npx prisma generate && npx prisma migrate deploy && npm run build`
- `/api/health` returns `{ ok: true }` (Render uses this as the health check)

## 1. Create a free Neon Postgres database

1. Log into [Neon](https://console.neon.tech) (sign up if needed — GitHub auth works).
2. **New Project**:
   - Name: `aisb-invoicer`
   - Postgres version: 16
   - Region: **AWS US East (Ohio)** — same region as the Render Web
     Service for lowest latency.
3. Neon creates the project + a default database called `neondb`. On
   the project dashboard, find the **Connection string** (with
   "Pooled connection" toggled ON for production traffic).
4. Copy the full URL — it looks like:
   ```
   postgresql://user:pass@ep-xxx-pooler.us-east-2.aws.neon.tech/neondb?sslmode=require
   ```
   Keep this tab open; you'll paste it into Render in step 3.

## 2. Apply the Render Blueprint

1. Log into [Render](https://dashboard.render.com).
2. **New** → **Blueprint**.
3. Connect the GitHub repo `JP-Billionz/aisb-invoicer`.
4. Render reads `render.yaml` and proposes one resource:
   - **Web Service**: `aisb-invoicer` (Node, **Free** plan, Ohio region)
5. Click **Apply**. Render creates the service but the first build will
   fail because `DATABASE_URL` isn't set yet — that's expected, fix it in
   step 3.

## 3. Set the three env vars that need a human

Open the Web Service → **Environment** tab and set:

| Key | Value |
|---|---|
| `DATABASE_URL` | The Neon pooled connection string from step 1 |
| `NEXTAUTH_URL` | `https://aisb-invoicer.onrender.com` (use whatever URL Render assigned). After a custom domain is wired in step 5, change to `https://invoices.aisolutionsbb.com` |
| `SENDGRID_API_KEY` | Leave blank for Phase 1A. Magic-link URLs print to the Render logs (see step 4). |

`NEXTAUTH_SECRET` was auto-generated by the Blueprint — no action needed.

Save → Render kicks off a fresh deploy. First build runs `prisma migrate
deploy` against Neon, creating all tables. Build takes ~5 minutes on
the free plan.

## 4. Smoke test the staging URL

Once the deploy is green:

```bash
# Health check
curl -fsS https://aisb-invoicer.onrender.com/api/health
# → {"ok":true,...}

# Marketing page
open https://aisb-invoicer.onrender.com
```

> **Free-plan note:** if the service has been idle 15+ minutes, the
> first request takes 30-60s to spin back up. Subsequent requests are
> fast. Phase 1B upgrades to Starter ($7/mo) which always-on.

**Sign-in flow (Phase 1A):**
1. Click "Sign in" → enter your email → submit.
2. Open Render dashboard → Web Service → **Logs** → look for:
   ```
   === MAGIC LINK (Phase 1A — no live email) ===
   To:   you@example.com
   Link: https://aisb-invoicer.onrender.com/api/auth/callback/email?...
   ```
3. Paste that URL into your browser. You're signed in. A tenant is
   auto-created with `nextInvoiceNumber = 10027` and a 14-day trial.

## 5. Custom domain (optional for staging, required for launch)

1. Render Web Service → **Settings** → **Custom Domains** → **Add Custom
   Domain** → `invoices.aisolutionsbb.com`.
2. Render shows a CNAME target like `aisb-invoicer.onrender.com`.
3. Cloudflare DNS → **Add record**:
   - Type: CNAME
   - Name: `invoices`
   - Target: the Render CNAME
   - Proxy: **DNS only** (gray cloud) until the SSL cert issues, then
     you can proxy it.
4. Wait ~5 minutes. Render auto-issues a Let's Encrypt cert.
5. Update `NEXTAUTH_URL` to `https://invoices.aisolutionsbb.com` in the
   Web Service env vars and redeploy.

## If something breaks

- **First build fails with "Can't reach database server"** → `DATABASE_URL`
  not set in Render env vars. Paste the Neon pooled connection string.
- **`prisma migrate deploy` fails with "Migration provider mismatch"**
  → the Neon DB has an existing schema from another project. Use a
  fresh Neon project, or run `npx prisma migrate reset` against it.
- **Build fails on `npm install`** → Render's Node version. We pin
  `engines.node >= 20` in `package.json`; Render should pick Node 20.
  If not, set `NODE_VERSION=20` in the env vars.
- **Sign-in returns 500** → almost always `NEXTAUTH_URL` mismatch with
  the actual host. Make sure it equals the URL in your browser bar.
- **Service won't spin up after idle** → free plan limitation. Hit
  `/api/health` first to wake it, then retry the page load.
